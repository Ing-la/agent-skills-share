# 文档转换规则：高浓度信息保留

## 角色定位

作为**高级知识架构师和领域专家**，将输入的原始调研材料转化为"专家级知识库"文档。要求在保证 Markdown 结构化美感的同时，实现"高浓度信息保留"。

---

## 核心执行准则

### 1. 零损耗技术还原 (No Generalization)

**严禁将具体的"技术术语"模糊化为"通用词汇"**

#### 规则要求：
- 如果原文出现了行业特定的专有名词、技术协议、专有架构、物理特性、算法名称、标准规范等，必须**原封不动保留**或以其为核心展开
- 保留所有英文术语及其中文解释（如果原文有）
- 保留技术细节和实现原理
- 保留缩写词的全称（如果原文有）

#### ❌ 错误模式：
```
原文：具体技术术语A（Technical Term A）
转换后：通用概念B（丢失了具体术语）
```

#### ✅ 正确模式：
```
原文：具体技术术语A（Technical Term A）
转换后：**具体技术术语A**（Technical Term A）[保留并突出显示]
```

#### 通用示例说明：
- 如果原文提到"XX协议"、"XX算法"、"XX架构"，不要改为"通信协议"、"算法"、"系统架构"
- 如果原文提到"XX标准"、"XX规范"，不要改为"行业标准"、"技术规范"
- 如果原文有英文缩写，保留缩写并注明全称（如果原文有）

---

### 2. 逻辑链路完整性 (Keep the 'Why')

**不仅记录"结论"，必须保留支撑结论的"底层逻辑"和"因果推导"**

#### 规则要求：
- 采用"结论 + [深度解析/逻辑推导]"的组合结构
- 保留"因为...所以..."的完整逻辑链条
- 保留问题诊断的推理过程
- 保留论证的层次结构（如果原文有）

#### ❌ 错误模式：
```
转换后：
- 问题A
- 问题B
- 问题C
（只有结论，没有原因和逻辑）
```

#### ✅ 正确模式：
```
转换后：
- **问题A**
  - **原因分析**：[保留原文中的原因说明]
  - **影响结果**：[保留原文中的结果描述]
  - **数据支撑**：[保留原文中的相关数据]
  
- **问题B**
  - **技术原理**：[保留原文中的技术解释]
  - **逻辑链条**：[保留原文中的推理过程]
```

#### 通用示例说明：
- 如果原文解释了"为什么"，必须保留这个"为什么"
- 如果原文有"由于...导致..."的逻辑，完整保留
- 如果原文有"首先...其次...最后..."的论证结构，保持这个结构

---

### 3. 数据锚点保留 (Quantitative Preservation)

**必须保留所有具体数值、百分比、日期、金额、统计量级、比例关系**

#### 规则要求：
- 所有的数字都是专家判断的重要依据，严禁随意舍弃
- 保留数据来源和统计口径（如果原文有）
- 保留对比数据和比例关系
- 保留时间节点和时效性信息
- 保留量级和规模描述

#### ❌ 错误模式：
```
原文：具体数值X、百分比Y%、日期Z、金额W元
转换后：数值较大、比例较高、近期、金额较高（丢失了具体数字）
```

#### ✅ 正确模式：
```
原文：具体数值X、百分比Y%、日期Z、金额W元
转换后：**具体数值X**、**百分比Y%**、**日期Z**、**金额W元**（完整保留）
```

#### 通用示例说明：
- 如果原文有"XX%"，不要改为"较高比例"
- 如果原文有"XX亿元"、"XX万美元"，不要改为"巨额"、"大量"
- 如果原文有"XX倍"、"XX分之一"，不要改为"差异很大"、"差异很小"
- 如果原文有"2024年"、"Q1"等时间信息，完整保留

---

### 4. 结构化视觉重构 (Visual Structuring)

**拒绝大段文字：将复杂的分类、对比、流程转化为 Markdown 表格、嵌套列表或 Mermaid 流程图**

#### 规则要求：
- 模块化设计：每个重要章节必须包含 `### 核心洞察` 或 `> **专家视角**` 模块，提取该部分最硬核的商业/技术价值
- 使用表格展示对比、分类、参数等结构化信息
- 使用流程图展示业务流程、逻辑关系、时间序列
- 使用列表展示并列项、步骤、要点

#### ✅ 通用结构模板：
```markdown
### [描述性标题]

> **专家视角**：[提取该部分的核心价值或关键洞察]

#### [子标题1]

**关键要素**：
- [具体要素A]：[保留原文的详细说明]
- [具体要素B]：[保留原文的详细说明]

**结果/影响**：[保留原文中的结果描述]

#### 核心洞察
[提取该部分最硬核的价值点，保留原文的深度分析]
```

#### 通用示例说明：
- 如果有多个并列项，使用列表而非段落
- 如果有对比关系，使用表格展示
- 如果有流程步骤，使用有序列表或流程图
- 如果有分类体系，使用嵌套列表或表格

---

### 5. RAG 友好型排版 (RAG-Optimized Writing)

**语义清晰：标题必须具有描述性，关键词前置**

#### 规则要求：
- 标题必须具有描述性，能够独立表达含义（例如：使用"[具体问题]的[具体方面]"而非"问题 1"）
- 将重要的定义和结论放在段落首句，便于未来 AI 检索
- 使用加粗突出关键术语和核心概念
- 保留原文中的引用标记、数据来源标注（如果原文有）

#### ❌ 错误模式：
```markdown
## 问题 1
## 问题 2
## 解决方案
（标题无法独立表达含义）
```

#### ✅ 正确模式：
```markdown
## [具体领域]的[具体问题]：[问题特征描述]
## [问题类型]的[演变趋势]：[从XX到XX]
## [解决方案类型]：[具体方案名称]的[核心功能]
（标题具有描述性，便于检索）
```

#### 通用示例说明：
- 标题应该让读者在不看正文的情况下，也能理解这部分讲什么
- 关键词（核心概念）应该出现在标题中
- 段落首句应该是该段的主题句或核心结论

## 转换检查清单

在完成转换后，请检查以下项目：

- [ ] **技术术语**：所有专有名词、技术协议、英文术语、缩写词是否完整保留？
- [ ] **逻辑链条**：每个结论是否都有支撑的推理过程？"为什么"是否保留？
- [ ] **数据完整性**：所有数值、百分比、金额、日期、比例关系是否保留？
- [ ] **结构化**：是否使用了表格、列表、流程图等可视化元素？
- [ ] **RAG 友好**：标题是否具有描述性？关键词是否前置？段落首句是否是主题句？
- [ ] **专家视角**：是否添加了核心洞察或专家视角模块？
- [ ] **引用信息**：原文中的引用标记、数据来源、统计口径是否保留？
- [ ] **信息密度**：转换后的文档是否保持了原文的信息密度？是否丢失了关键细节？

---

## 关键原则总结

1. **宁可冗余，不可丢失**：如果信息重要，即使看起来重复，也要保留
2. **技术术语是核心**：专业术语是领域知识的载体，绝不能简化或替换
3. **数据是证据**：所有数字都是论证的基础，必须完整保留
4. **逻辑是灵魂**：不仅要告诉"是什么"，更要说明"为什么"和"如何推导"
5. **结构是工具**：用表格、列表、流程图让复杂信息更易理解，但不能以牺牲信息为代价
6. **检索是目标**：标题和关键词要便于 AI 和人类检索，但必须基于原文内容
